Vision shift

What changed from v1

The v1 plan was a 4-package, 30-week suite covering viz, analysis, reports, and dashboards. After critical review, scope collapsed to where the genuine value-add lives: opinionated themes and the hard-to-build consulting chart types. Everything else is either solved by existing tools or out of scope for one developer.

License trajectory

ggplot2, scales, officer, systemfonts are MIT. rvg (which gives editable PPT vectors) is GPL-3. ctplot v0.1 uses officer-only and ships rasterized charts → MIT is fine. When ctplot v1.0 adds rvg for editable vector charts, the package becomes GPL-3. Communicate this trajectory upfront in README.

Package 01

ggconsulting

Opinionated ggplot2 extension for executive-grade output. Three archetypes capture ~80% of corporate consulting visual identity. Builder API for the rest. Locale-aware labels. Font management without footguns.

Three archetypes

Conservative business visual identities collapse into three families. Function names are generic; documentation hints at firm-style inspiration without naming.

theme_strategy()

Minimal, generous whitespace, sans-serif, navy default. Inspired by top-tier global strategy consultancies.

main_color arg covers most variants in this family.

theme_finance()

Denser, subtle gridlines, serif body, classical typography. Suits Big 4 transaction services, bank advisory.

Designed to sit comfortably in financial reports and pitch books.

theme_editorial()

Warmer palette, italics in subtitles, typographic personality. Economist/FT/Bloomberg-style analyst note look.

For client-facing memos and market commentary, not transactional decks.

The ct_theme() builder

Underlying flexible API. The three archetypes are pre-set paths through this builder.

# Flexible builder — every theme arg exposed
ct_theme(
  palette     = "navy",         # named palette or hex vector
  font        = "Inter",        # with fallback chain
  density     = "normal",       # "tight" | "normal" | "loose"
  context     = "presentation", # "presentation" | "report" | "screen"
  base_size   = 12
)

# Archetypes are presets over the builder
theme_strategy(main_color = "navy")
theme_finance()
theme_editorial()

Context-aware sizing

Single theme, three contexts. Avoids parallel theme_strategy_pdf()/theme_strategy_html() families.

presentation
base_size 14, generous plot.margin, bold titles, high-contrast gridlines (16:9 slides)
report
base_size 10, tight margins, lighter gridlines (Quarto body text, A4/Letter PDF)
screen
base_size 11, default ggplot2-like behavior (interactive exploration)

Scales and palettes

  • scale_color_ct() / scale_fill_ct() — discrete + continuous variants
  • ~6 palettes total: 3 archetype defaults + 3 CVD-safe variants, tagged in docs
  • Categorical palette behavior: interpolate + warn when n > palette size (not silent cycle)
  • ct_palette_show() for quick visual preview

Geom defaults — autoload pattern

The systematic ggplot2 fixes (line width, column width, expansion) ship as global defaults applied at library() time. Loud message + opt-out path.

# On library(ggconsulting):
ℹ ggconsulting set ggplot2 geom defaults (lwd, col width, ...)
i Opt out: ct_unset_defaults() or options(ggconsulting.autoload = FALSE)

# User controls
ct_set_defaults()    # reapply
ct_unset_defaults()  # revert to pre-load state (not ggplot2 baseline)

Known risk

Autoload changes global ggplot2 state, affecting plots that didn't ask for the theme. Honors options(ggconsulting.autoload = FALSE) set in .Rprofile. If user feedback is hostile, flip default to opt-in.

ct_finish() — data-aware polish

Complements theme() by doing what theme can't: post-hoc, data-aware finishing. Sort, label, format, highlight, set scale expansion — all after the geom layer is built. Auto-detects the geom to pick sensible expansion.

ggplot(d, aes(category, value)) +
  geom_col() +
  ct_finish(
    values      = TRUE,        # value labels above bars
    sort        = "desc",      # reorder by value
    label_fmt   = "brl",       # locale formatter
    highlight   = "Acme",      # de-emphasize others
    end_labels  = TRUE,        # for line charts
    expand      = "auto"       # geom-aware scale expansion
  )

Scale expansion is handled inside ct_finish() rather than via separate ct_scale_*() wrappers. Auto-detection: geom_col → expansion(mult = c(0, 0.15)) on y (room for value labels above bars); geom_line with dates → right-side x expansion for end labels; etc. Users can override per-call.

Locale

Brazilian formatting is the default; US fallback always available. Does not rely on Sys.setlocale() (breaks on Windows CI).

Font strategy

Assume installed + helpers. Modern tooling (systemfonts, ragg-aligned). No font shipping in the package.

  • install_consulting_fonts() downloads Google Fonts (Inter, Source Sans 3, Lato, Source Serif 4, IBM Plex Sans — all OFL/Apache)
  • has_font(name) — for graceful test/example skipping
  • Fallback chain in every theme: c("Inter", "Helvetica Neue", "Arial", "sans")
  • Document showtext vs ragg gotchas in a vignette

Dependencies (Imports)

Ruthlessly minimal. Faster CRAN checks, fewer install failures.

Package 02

ctplot

The hard charts ggplot2 can't express cleanly + a PPT export path. Deliberately small surface — three constructors, one export function. Imports ggconsulting for defaults; works without it but looks plain.

Core constructors

FunctionPurposeWhy it earns its keep
ct_waterfall() McKinsey-style bridge chart with connectors and totals Genuinely hard in ggplot — running sums, connector lines, total bars, value labels above/below by sign
ct_slope() Before/after comparison with category labels at both ends Used heavily in consulting; ggplot can do it but takes ~30 lines per chart
ct_dumbbell() Gap analysis with two points + connecting segment Two geom layers + manual coord_flip + labels — wrapper saves real time

PPT export

Two levels, sequenced. Honest about what works.

Level 1 — v0.1

Single chart → PPTX from scratch

  • Raster PNG embedded in new slide
  • Title + subtitle text in slide layout
  • officer only — MIT-compatible
  • ~1 week of work
Level 2 — v0.x target

Chart → existing template slide

  • Drop chart into slide N of branded deck
  • Match placeholder layout (size, position)
  • Still raster — editable vectors deferred to v1.0
  • ~3–4 weeks of work

Deferred: editable vector PPT (rvg)

Promised "editable vector" charts only work cleanly for plain ggplots. Waterfall, marimekko, ggtext annotations break the rvg pipeline. Deferred to ctplot v1.0 with documented caveats — only after Level 2 has real users.

Out of public scope — ekioplot

The simple wrappers (ct_bar, ct_line, ct_scatter) live in ekioplot — a personal/company package named after Ekio. GitHub-only, no CRAN discipline, evolves fast. Imports ggconsulting and (later) ctplot. ctplot public surface stays small and disciplined.

Cross-cutting

Decisions that span both packages

Naming

  • ggconsulting — follows gg* extension convention (ggthemes, ggtext, ggrepel). Self-describing.
  • ctplot — short prefix, no collision with base::chartr()
  • Action before scaffolding: run available::available("ggconsulting") and available::available("ctplot") to check CRAN, Bioconductor, GitHub, and rude-word lists

Trademark and visual identity

  • Generic function names — never theme_bcg()
  • Docs use phrases like "inspired by top-tier global strategy consultancies"
  • Vignettes use a fake firm ("Acme Strategy Partners") with realistic chart titles
  • README does not name real firms in the body; one-line hints OK in vignettes
  • No real firm logos, ever
  • No proprietary fonts shipped — only Google Fonts free alternatives

Reference collection

  • Location: data-raw/references/ + listed in .Rbuildignore (not shipped to CRAN)
  • Per-reference fields: source URL, observed hex colors, observed/approximated font, spacing notes, geom used, annotation style
  • Target: 30–50 references across 3 archetypes
  • When reproducing: change data and titles; never use logos verbatim

Testing

  • vdiffr for visual regression — but font-dependent snapshots must skip_on_cran()
  • Two-tier baseline: layout baseline (CRAN-runnable, base_family = "") + font baseline (local-only)
  • Wrap font-styled tests in skip_if_not(has_font("Inter"))
  • Budget ~20% of dev time for tests

Versioning and lifecycle

  • Start at 0.0.0.9000 in DESCRIPTION (tidyverse dev convention)
  • Use lifecycle badges: ct_finish() and ct_theme() builder as experimental
  • Stable from day one: archetype themes, scale_color_ct/scale_fill_ct

Documentation

  • README with visual gallery (3 archetypes × 2 chart types = 6 example plots)
  • Vignette: "Choosing a theme" — side-by-side comparisons
  • Vignette: "Locale and number formatting"
  • Vignette: "Fonts: installing, fallbacks, ragg vs showtext"
  • pkgdown reference page with archetype gallery
  • Doc effort ≈ 15% of total

Roadmap

ggconsulting v0.1 — ~10 weeks

Single developer, part-time. Calendar time may be ~12 weeks with normal life buffer. Hard gate before ctplot: ship ggconsulting and dogfood it on at least one real consulting deck.

Week 1 — Phase 0

Scaffolding and final calls

usethis::create_package(). Naming availability check. License files. Confirm font fallback chain. Resolve open questions (scale expansion, negative number format). Write one-page design summary in repo.

Weeks 2–4 — Phase 1

Theme builder + 3 archetypes

ct_theme() builder with all args. Three archetype themes as preset paths. base_size and context sizing logic. Initial vdiffr baselines (layout only).

Weeks 5–6 — Phase 2

Palettes and scales

6 palettes (3 archetype defaults + 3 CVD-safe). scale_color_ct() / scale_fill_ct() with discrete + continuous. Interpolation behavior with warning. ct_palette_show().

Week 7 — Phase 3

Geom defaults and locale

update_geom_defaults() on load + opt-out machinery. Brazilian + US number formatters. ct_locale(). Internal Portuguese month tables. ct_finish() initial implementation.

Week 8 — Phase 4

Fonts and references

install_consulting_fonts(). has_font(). Test font registration paths on macOS/Linux/Windows. Build reference catalog in data-raw/references/.

Weeks 9–10 — Phase 5

Tests, docs, polish, CRAN prep

vdiffr suite with two-tier baselines. Three vignettes. pkgdown site with gallery. README with example plots. R CMD check --as-cran clean. Optionally submit to CRAN.

ctplot v0.1 — after ggconsulting

Estimated ~6–8 weeks. ct_waterfall + ct_slope + ct_dumbbell + ct_to_pptx() Level 1. Heavily dogfooded against real decks. Level 2 deferred to v0.2.

Repo layout

Two options, each with tradeoffs:

  • Separate repos — clean CI per package, independent release cadence, slower context-switching across packages
  • Monorepo (e.g., consultr/ggconsulting, consultr/ctplot as sibling dirs) — faster cross-package edits, but harder to onboard contributors and ship to CRAN

Suggested: separate repos. CRAN expects one package per repo; release independence matters more than edit speed.

Success criteria

How we'll know v0.1 worked

A single dogfooding criterion. Stars, downloads, and CRAN status follow from real use; they don't substitute for it.

v0.1 success

I use ggconsulting on three real consulting decks within a month of v0.1 release, and don't need to monkey-patch anything to make it work.

If monkey-patching is needed, the API is wrong — fix it before announcing. If three decks happen smoothly, ggconsulting has earned a wider release and ctplot scaffolding can start.
